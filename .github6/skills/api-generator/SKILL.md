# API Generator Skill

Generate backend API routes, services, and middleware from an API specification document.

## When to Use

Use this skill when you need to:
- Generate route handlers from `docs/api-spec.md`
- Create service layer with business logic
- Implement authentication and authorization middleware
- Set up request validation and error handling

## Inputs

This skill requires:
1. `docs/api-spec.md` — Endpoint definitions, request/response schemas, auth requirements
2. `docs/architecture.md` — Backend framework choice and project structure
3. `docs/data-model.md` — Entity definitions for service layer logic
4. Database schema (generated by the data-modeling skill)

## Process

### Step 1: Analyze API Spec
- Parse all endpoint groups from `docs/api-spec.md`
- Catalog HTTP methods, paths, auth requirements, and roles
- Identify request/response shapes and validation rules
- Note shared patterns (pagination, filtering, sorting)

### Step 2: Generate Middleware
Create cross-cutting middleware:

**Authentication middleware:**
- Token/session verification
- User extraction and attachment to request context
- Unauthenticated request rejection

**Authorization middleware:**
- Role-based access control factory (parameterized by allowed roles)
- Permission checking

**Validation middleware:**
- Request body validation using a schema library (Zod, Joi, etc.)
- One validation schema per endpoint
- Consistent error format on validation failure

**Error handling middleware:**
- Global error handler
- Consistent error response format: `{ error: { code, message, details? } }`
- Development vs. production error detail levels

### Step 3: Generate Services
For each resource/domain area:

- Create a service file with business logic functions
- Each function handles one operation (create, read, update, delete, transition)
- Services call the data layer (ORM/repository) — not the database directly in routes
- Enforce business rules and constraints from the app spec
- Use database transactions for multi-write operations
- Throw typed errors that the error middleware can handle

### Step 4: Generate Route Handlers
For each endpoint in `docs/api-spec.md`:

- Create thin route handler that:
  1. Extracts parameters from request (params, query, body)
  2. Calls the appropriate service function
  3. Returns the response with correct status code
- Apply middleware: auth → authorize → validate → handler
- Group routes by resource

### Step 5: Generate App Entry Point
- Wire all route groups into the application
- Apply global middleware (CORS, request logging, body parsing, security headers)
- Configure error handling
- Export the app for both running and testing

### Step 6: Validate
- Verify every endpoint in `docs/api-spec.md` has a route handler
- Verify authentication/authorization requirements match the spec
- Verify request validation schemas match the spec
- Verify response shapes match the spec

## Output

| Directory/File | Contents |
|----------------|----------|
| `src/middleware/` | Auth, validation, error handling middleware |
| `src/services/` | Business logic per resource |
| `src/routes/` | Route handlers per resource |
| `src/validators/` | Request validation schemas |
| `src/app.ts` or `src/index.ts` | App entry point and configuration |

## Conventions

- Thin handlers — all logic lives in services
- Never import the database directly in route handlers
- Async/await with proper error propagation
- Consistent response shapes across all endpoints
- Log meaningful events (auth failures, errors) — not raw request data
- Follow security instruction file conventions
